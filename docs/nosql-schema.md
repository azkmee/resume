# Resume Website — MongoDB Schema Plan

> Data model for a personal resume + portfolio + blog website.
> Content is authored by a human **and** generated by Claude, then written to MongoDB
> through the [MongoDB MCP server](https://github.com/mongodb-js/mongodb-mcp-server).
>
> Source material for the resume side lives in [`../accomplishments-log.md`](../accomplishments-log.md)
> (STAR-format wins) and is migrated into the `accomplishments` collection.

---

## 1. Design goals

1. **Read-optimized for rendering.** The website reads far more than it writes. Documents
   are shaped so a page can be rendered from one or two queries, not a web of joins.
2. **Author-once, review-then-publish.** Every content document carries a `status`
   (`draft → in_review → published → archived`) and `provenance` so Claude-generated drafts
   are clearly distinguishable from reviewed, human-approved content.
3. **Idempotent writes.** Every content collection has a stable, human-readable `slug`.
   Writes are `upsert`-by-slug so re-running a generation never creates duplicates.
4. **Schema-validated.** Each collection has a `$jsonSchema` validator (see [`../db/schema/`](../db/schema)).
   Malformed documents — including ones Claude might hallucinate — are rejected at write time.
5. **Storage-agnostic media.** Images, diagrams, and files live behind a `media` collection
   that abstracts *where* the bytes are (R2 / S3 / Cloudinary / GridFS / inline). Blog and
   project documents reference media by id, never by raw storage path.

---

## 2. Collections at a glance

| Collection      | Cardinality        | Purpose |
|-----------------|--------------------|---------|
| `profile`       | singleton          | You: name, headline, summary, contacts, socials, current role |
| `experiences`   | ~5–15              | Jobs / roles (the `## Role:` sections in the log) |
| `accomplishments` | dozens+          | STAR wins, tags, metrics, resume-bullet drafts — references an experience |
| `projects`      | ~10–50             | Portfolio projects with tech stack, links, gallery |
| `skills`        | ~30–80             | Skill inventory, grouped by category, with proficiency |
| `education`     | ~2–10              | Degrees & certificates |
| `blog_posts`    | growing            | Articles with a **block-based body** (text / image / diagram / html / code) |
| `media`         | growing            | Asset registry: images, diagrams, files — storage-abstracted |
| `pages`         | ~5–20              | Flexible block-based content for arbitrary sections (about, now, uses, contact) |
| `testimonials`  | ~5–30              | Recommendations / quotes from colleagues |

**Why these splits?**

- `accomplishments` is its own collection (not embedded in `experiences`) because there are
  many of them, they're queried/filtered independently (by tag, by date), and Claude appends
  to them continuously. Embedding would mean rewriting a large role document on every new win.
- `media` is centralized so one image can be reused across posts/projects, and so a storage
  migration (e.g. GridFS → R2) touches one collection, not every document that shows a picture.
- `pages` uses the **same block model as `blog_posts`**, so "other things I want to show on the
  site" don't each need a bespoke schema — they're just blocks.

## 3. The common envelope

Every **content** document (everything except `profile` and `media`) shares this envelope.
Collection-specific fields are added on top.

```jsonc
{
  "_id": "ObjectId",
  "slug": "software-engineering-lead-esgpedia",  // url-safe, unique within collection
  "status": "draft",            // draft | in_review | published | archived
  "visibility": "public",       // public | unlisted | private
  "featured": false,            // surface on home / highlight reels
  "order": 0,                   // manual sort within a section (lower = earlier)
  "tags": ["leadership", "backend"],

  "seo": {                      // optional; used to render <head>
    "title": "…",
    "description": "…",
    "keywords": ["…"],
    "ogImageId": "ObjectId|null",   // -> media._id
    "canonicalUrl": "https://…|null"
  },

  "provenance": {               // where this content came from — key for the Claude workflow
    "source": "claude",         // claude | human | import
    "model": "claude-opus-4-8", // when source = claude
    "sourcePrompt": "…|null",   // optional: the prompt that produced it
    "sourceRef": "accomplishments-log.md#2026-04-stepped-into-lead",  // optional provenance link
    "generatedAt": "ISODate|null",
    "reviewedBy": "human|null", // null until a human approves
    "reviewedAt": "ISODate|null"
  },

  "createdAt": "ISODate",
  "updatedAt": "ISODate",
  "publishedAt": "ISODate|null",  // set when status -> published
  "version": 1                    // bump on each meaningful edit
}
```

**Publishing rule of thumb:** the website's public queries always filter
`{ status: "published", visibility: "public" }`. Drafts are invisible to visitors until a
human flips the status — Claude can write freely without anything leaking to production.

### 4.5 `skills`

One document per skill (queryable, sortable). Group at render time with a simple aggregation.

```jsonc
{
  "_id": "ObjectId",
  "category": "Server-side / Frameworks", // Languages | Client-side | Server-side/Frameworks | Data/Infra | DB tooling | Testing/DevOps | Domain | Leadership/Process
  "name": "Spring Boot",
  "level": 5,                  // 1–5 (or null)
  "years": 5,                  // optional
  "featured": true,
  "order": 0,
  "updatedAt": "ISODate"
}
```

> Render-time grouping:
> ```js
> db.skills.aggregate([
>   { $sort: { order: 1, name: 1 } },
>   { $group: { _id: "$category", items: { $push: { name: "$name", level: "$level", featured: "$featured" } } } }
> ])
> ```

### 4.7 `blog_posts` — the block-based body

This is where **pictures, diagrams, and HTML** are handled cleanly. The body is an ordered
array of typed **blocks**. A renderer walks the array and renders each block by `type`.

```jsonc
{
  "_id": "ObjectId",
  "slug": "building-an-ai-in-repo-knowledge-base",
  "title": "Building an AI-driven in-repo knowledge base",
  "excerpt": "Why we moved docs out of stale Confluence and into the codebase.",
  "coverMediaId": "ObjectId|null",
  "author": "profile",                 // -> profile._id
  "readingTimeMinutes": 6,
  "wordCount": 1200,
  "series": null,                      // { "id": "ai-tooling", "title": "AI Tooling", "part": 2 } | null
  "relatedPostIds": ["ObjectId"],
  "body": {
    "format": "blocks",
    "blocks": [
      { "key": "b1", "type": "markdown", "order": 0, "text": "## The problem\nConfluence went stale…" },

      { "key": "b2", "type": "image", "order": 1,
        "mediaId": "ObjectId", "alt": "Architecture before", "caption": "The old flow", "width": 1200, "height": 700 },

      { "key": "b3", "type": "diagram", "order": 2,
        "engine": "mermaid",            // mermaid | excalidraw | svg
        "source": "graph TD; A[Repo]-->B[README]; B-->C[Claude]; C-->B;",
        "renderedMediaId": "ObjectId|null",  // optional pre-rendered PNG/SVG for no-JS clients
        "caption": "Generation loop" },

      { "key": "b4", "type": "code", "order": 3,
        "language": "js", "filename": "setup.mongosh.js", "code": "db.createCollection(/*…*/)" },

      { "key": "b5", "type": "html", "order": 4,
        "html": "<iframe src='https://…'></iframe>", "sanitized": false },  // raw HTML embed

      { "key": "b6", "type": "callout", "order": 5, "variant": "info", "text": "Tip: …" },

      { "key": "b7", "type": "gallery", "order": 6,
        "items": [ { "mediaId": "ObjectId", "alt": "…", "caption": "…" } ] },

      { "key": "b8", "type": "embed", "order": 7, "provider": "youtube", "url": "https://youtu.be/…" },

      { "key": "b9", "type": "quote", "order": 8, "text": "…", "cite": "Someone" },

      { "key": "b10", "type": "divider", "order": 9 }
    ]
  },
  // + common envelope (publishedAt drives the blog index ordering)
}
```

**Supported block types:** `markdown`, `html`, `image`, `gallery`, `diagram`, `code`,
`callout`, `embed`, `quote`, `divider`. Add new types by extending the renderer + validator —
existing posts are unaffected.

> **Security note on `html` blocks:** raw HTML from a generator is an injection risk. Either set
> `sanitized: true` after server-side sanitization (e.g. DOMPurify), or restrict `html` blocks to
> `provenance.source: "human"`. Prefer `embed`/`diagram` blocks over raw `html` where possible.

### 4.8 `media` — storage-abstracted asset registry

```jsonc
{
  "_id": "ObjectId",
  "kind": "image",                // image | diagram | file | video
  "storage": "r2",                // r2 | s3 | cloudinary | gridfs | inline
  "url": "https://cdn.example.com/blog/arch-before.webp",  // public URL (r2/s3/cloudinary)
  "key": "blog/arch-before.webp", // object key (r2/s3); unique
  "gridfsId": null,               // -> fs.files._id when storage = gridfs
  "inline": null,                 // { encoding: "svg"|"base64", data: "…" } when storage = inline
  "mime": "image/webp",
  "bytes": 84210,
  "width": 1200,
  "height": 700,
  "alt": "Architecture before",   // default alt; blocks can override
  "blurhash": "L6Pj0^…",          // optional LQIP
  "checksum": "sha256:…",         // dedupe
  "diagramSource": null,          // { engine: "mermaid", source: "…" } when this asset is a rendered diagram
  "tags": ["architecture"],
  "createdAt": "ISODate",
  "updatedAt": "ISODate"
}
```

> **Cost-driven default (see plan §0):** raster images → `storage: "r2"` (zero egress);
> small diagrams → either `storage: "inline"` (SVG/Mermaid text in the doc) or kept as
> `diagramSource` on the block itself. GridFS is supported but discouraged for a public site.

## 5. Indexes

See [`../db/indexes.json`](../db/indexes.json) for the machine-readable definitions applied by the
setup script. Highlights:

- **Unique slugs** per content collection (`projects.slug`, `blog_posts.slug`, …) → idempotent upserts.
- **Publish queries:** `{ status: 1, publishedAt: -1 }` on `blog_posts`; `{ status: 1, visibility: 1 }` broadly.
- **Tag filters:** multikey `{ tags: 1 }`.
- **Accomplishments:** `{ experienceId: 1, "date.iso": -1 }` and a **text index** across title + STAR fields
  + resume bullets for search.
- **Media:** unique sparse `{ key: 1 }`, plus `{ checksum: 1 }` for dedupe.

## 6. The Claude → MCP authoring workflow

```
┌────────────┐   generates    ┌──────────────────┐   upsert by slug   ┌────────────┐
│   Claude   │ ─────────────▶ │  doc matching     │ ─────────────────▶ │  MongoDB   │
│  (in chat) │   draft doc    │  the $jsonSchema   │   via Mongo MCP    │  (validated)│
└────────────┘                └──────────────────┘                     └─────┬──────┘
                                                                              │
                            human reviews ◀── status: "draft"/"in_review" ────┘
                                   │
                                   ▼  set status:"published", provenance.reviewedBy:"human"
                              visible on the website
```

**Rules that make this safe and repeatable:**

1. **Always write drafts.** Claude-generated docs are created with
   `status: "draft"`, `provenance.source: "claude"`, `provenance.model`, and `generatedAt`.
   Public site queries never show drafts, so nothing leaks before review.
2. **Upsert by `slug`, never blind insert.** Use
   `updateOne({ slug }, { $setOnInsert: { createdAt, version: 1 }, $set: { …fields, updatedAt } }, { upsert: true })`.
   Re-running a generation updates in place instead of duplicating.
3. **Validators are the guardrail.** Because each collection has a `$jsonSchema`, a malformed
   Claude document is **rejected by the database**, not silently stored. Treat a validation error
   as a signal to fix the generated shape.
4. **Carry provenance.** Set `provenance.sourceRef` to the origin (e.g. an `accomplishments-log.md`
   anchor) and `provenance.sourcePrompt` to the prompt when useful. This makes every published
   sentence traceable back to its source.
5. **Human flips the switch.** Only a human sets `status: "published"` +
   `provenance.reviewedBy: "human"` + `reviewedAt`. Claude can suggest, never publish.

**MongoDB MCP server tools** (from `mongodb-mcp-server`) you'll use: `find`, `aggregate`,
`insert-many` / `update-many` (for upserts), `count`, and `collection-schema`. Point it at your
database with a connection string; restrict it to this database. Keep it in **read+write** mode
only for the authoring database, and consider a separate read-only connection for anything
production-facing.

## 8. Suggested rollout

1. Run `setup.mongosh.js` against a fresh database → collections + validators + indexes exist.
2. Run `migrate-accomplishments.mjs` → regenerates the experiences/accomplishments seed from the log.
3. Insert seed files via the Mongo MCP server (or `mongoimport`) → real data, all as `needs_review`.
4. Review & publish a first slice (profile, experiences, a couple of accomplishments).
5. Point the website at `{ status: "published", visibility: "public" }` queries.
6. From then on: capture wins in `accomplishments-log.md` as today **or** have Claude write straight
   to the `accomplishments` collection as drafts — same review-then-publish loop either way.
```
